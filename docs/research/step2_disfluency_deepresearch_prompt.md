# 役割と目的

あなたは音声言語処理（spoken language processing）と日本語自然言語処理の実務・調査に長けた専門リサーチャーです。
**日本語の自発音声（特に就職面接スピーチ）の文字起こしテキストから、「意味のある内容部分」と
「フィラー・言い淀みなどの非流暢性（disfluency）」を、アノテーションして分離し、定量化・解析可能に
する手法**について、実際の事例や実装可能な OSS/モデル/API を調査し、そのソースや概要とともに報告してください。

# 背景（このプロジェクトの文脈・前提知識として使ってよい）

- 動画面接スコアリング製品。OpenAI Whisper で文字起こしし、(1) 話し方（フィラー率・言い淀み等）と
  (2) 内容/構成 を評価する。
- Whisper は既定でフィラーを除去・正規化（補完）するが、prompt 誘導で verbatim（忠実）な文字起こしが
  得られることを実機検証済み。例（実データ verbatim）:
  「えーと、研究室では、えーと、…そこでは、えーと、シミュレテッドアニーリング、漁師、漁師、えーと、
   漁師なんとかシミュレーションを、えーと、専門で行っておりました。」
  - ここには (a) 語彙的フィラー（えーと/えー/うーん）と (b) 繰り返し・言い直し（漁師、漁師、漁師なんとか）
    が混在する。
- ゴール: verbatim を「内容語の層」と「disfluency の層」に分離・タグ付けし、フィラー率等を決定論的に
  定量化しつつ、内容評価にも使える中間表現を作りたい。LLM に内容を“捏造補完”させずに行いたい。

# すでに把握している事（重複説明は不要。これらを「裏取り・深掘り・反証」してほしい）

- disfluency の構造モデル: Shriberg 系の reparandum / interruption point / interregnum / repair。
- 日本語のデファクト標準: 国立国語研究所『日本語話し言葉コーパス(CSJ)』のタグ（(F)フィラー・(D)語断片・
  (D2)・(W)）、UniDic の品詞「感動詞-フィラー」。
- 手法の大別: 形態素解析ベース / 系列ラベリング(BERT等) / LLM ベース。

# 調査してほしい論点（各論点で参照できる実際の事例やソース情報があれば付す）

1. **disfluency の理論モデルと標準分類**
   - Shriberg (1994) の定義や一般的な考え方、および Switchboard 等での分類方法（reparandum/interregnum/repair などの定義）。
   - filled pause / repetition / revision(repair) / false start / restart の分類とその関係。

2. **日本語のアノテーション体系（実際の仕様）**
   - CSJ の非流暢性タグ仕様（(F)/(D)/(D2)/(W) などの定義と適用範囲）。
   - UniDic の品詞体系（「感動詞-フィラー」の定義や unidic / unidic-lite の扱い）。
   - CSJ 以外に日本語 disfluency の公開コーパス/タグ体系があれば（例: 名大会話コーパス, JTalk 等）列挙。

3. **自動検出手法の具体（実装可能性とベンチマーク）**
   - **形態素解析**: MeCab / fugashi / SudachiPy / GiNZA(spaCy-ja) それぞれで「感動詞-フィラー」相当が
     どこまで取れるか。各ライブラリ＋辞書（unidic, unidic-lite, SudachiDict）の対応状況・既知の限界。
   - **系列ラベリング(disfluency detection)**: 一般的な手法や日本語 CSJ での研究・F1 数値の事例。
     **HuggingFace 等に日本語の事前学習済み disfluency detection モデルが実在するか**（あればモデル名・URL・
     ライセンス・性能。無ければ「無い」と明記）。
   - **LLM ベース**: GPT-4o / Claude 等で disfluency 抽出・除去を行う実際の事例や技術記事と、
     報告されている精度・既知の失敗（特に「除去時に内容を捏造補完する」問題）と対策。
   - 各手法に **推奨度（採用候補/参考/非推奨）** と、その根拠（精度・実装/保守コスト・本番運用適性）。

4. **アノテーション/中間表現の設計（実例ベース）**
   - 標off注釈(standoff)・トークン単位 BIO/IOB2・span+type・「clean 層＋disfluency 層」二層表現など、
     研究や実システムで使われている表現方式と、それぞれの利点/欠点。可能なら実在のスキーマ例・データ形式。
   - **LLM のハルシネーション（文字 span の数え間違い・内容の捏造）を避ける実務的手法**：deletion-only 制約、
     アライメント（verbatim と clean の整合）、抽出のみ＋ロジック側で指標化、等の実際の事例やベストプラクティス。

5. **機能語の曖昧性解消（その/あの/まあ/なんか/こう 等）**
   - これらがフィラーか正規用法（指示詞/副詞）かを判別するアプローチや精度（実際の事例や手法）。
     形態素解析単独の限界を示す具体例と、文脈モデル（BERT/LLM）での改善の実証。

6. **本ユースケースへの推奨アーキテクチャ（結論）**
   - 「決定論（形態素）でフィラー率ベースライン ＋ LLM で複雑な disfluency 抽出と内容評価」という二段構えの
     妥当性を、精度・コスト・保守性の観点で評価。代替案（系列ラベリングの自己ホスト等）との比較表。
   - フィラー率の**指標定義の推奨**（per minute / per 文節 / per モーラ 等、日本語で頑健な分母は何か。
     CPM=文字/分 が漢字比率でブレる問題への対処を含む）。一般的な指標定義や先行事例を参考にする。
   - 想定される落とし穴と、最小実装で踏むべき手順の提案。
