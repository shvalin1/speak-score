# 面接スピーチにおける非流暢性（Disfluency）のアノテーションと解析手法の調査

面接動画評価システムにおいて、Whisperのverbatim（忠実な文字起こし）結果からフィラーや言い淀み（非流暢性：Disfluency）を分離・定量化し、内容評価用の中間表現を生成するための調査結果を報告します。

---

## 1. Disfluencyの言語学/NLP構造モデル
【確認済み】
自然言語処理（NLP）および音声言語処理におけるDisfluency（非流暢性）の標準的な構造モデルは、Shriberg (1994) などの研究に端を発し、以下の枠組みで整理されます。

発話の修正過程は、主に以下の3つの構成要素から成ります：
`[ Reparandum + Interruption Point (IP) + {Interregnum} + Repair ]`
*   **Reparandum（修復部）**: 言い間違えたり、意図が変わったりして、最終的に取り消される（破棄される）発話部分。
*   **Interruption Point (IP)**: 発話が中断された地点。
*   **Interregnum（編集段階）**: 中断から言い直し（修復）が始まるまでの間に挟まる部分。ここに「えーと」「あのー」などのフィラー（Filled pause）が含まれます。
*   **Repair（修復結果）**: 修復され、最終的に話し手が意図した発話部分。

**分類体系（標準的なカテゴリ）**
*   **Filled pause (フィラー)**: 「えーと」「あのー」などの無意味な音や場つなぎ。
*   **Repetition (繰り返し)**: 「私は、私は」のように同じ語を繰り返す現象。
*   **Repair / False start (言い直し/再出発)**: 「シミュレテッド…いや、量子シミュレーション」のように言い間違えを訂正する現象。
*   **Restart**: 発話を完全に途中で放棄し、新しい文を始める現象。

## 2. 日本語固有のアノテーション体系
【確認済み】
日本語の話し言葉における事実上の標準（デファクトスタンダード）は、国立国語研究所が構築した**『日本語話し言葉コーパス』（CSJ: Corpus of Spontaneous Japanese, 2004年公開）**です。

**CSJにおける非流暢性のタグ付け体系**
CSJではテキスト化の際、非流暢性を解析可能にするため以下のタグが付与されています。
*   **(F)**: フィラー、感情表出系感動詞。（例：`(F あの)`、`(F えーと)`）
*   **(D)**: 語断片。途中で途切れた不完全な語。（例：`(D ぎょ) 漁師`）
*   **(D2)**: 助詞、助動詞、接辞の言い直しや繰り返し。（例：`(D2 が)`）
*   **(W)**: 言い間違いなど。

**UniDicにおける品詞『感動詞-フィラー』の扱い**
*   形態素解析用の電子化辞書「UniDic」では、品詞体系として**「感動詞-一般」**（驚き、挨拶など）と**「感動詞-フィラー」**（発話の間の場つなぎ）が明確に区別して定義されています。
*   MeCab等の解析器にUniDicを組み合わせて用いることで、「えーと」「あのー」などは標準で「感動詞-フィラー」としてタグ付けされます。

## 3. 自動検出・分離の手法

### (i) 形態素解析ベース (MeCab/Sudachi + UniDic)
*   **精度・特徴**: 「えーと」「あのー」といった語彙的なフィラーは、UniDicの「感動詞-フィラー」タグを利用することで高速かつ高精度に抽出可能です。しかし、「私は、私は」のような繰り返し（Repetition）や、文脈に応じた言い直し（Repair）、語断片は形態素単体の辞書マッチングでは検出できません。
*   **実装難度**: 非常に低い（Python + MeCab/fugashi等で即時実装可能）。
*   **推奨度**: **【採用候補（部分的）】** - 軽量なフィラー率のベースライン計測用として。

### (ii) 系列ラベリング (BERT等によるDisfluency Detection)
*   **精度・特徴**: CSJデータセットなどでBERTをファインチューニングし、テキストの各トークンがDisfluency（修復部やフィラー）に該当するかをIOB2タグなどで系列ラベリング（Token Classification）する手法です（東北大学の研究などで実証済み）。文脈を考慮するため、繰り返しや言い直しも検出できます。
*   **実装難度**: 高。Hugging Faceの日本語モデル（`cl-tohoku/bert-base-japanese`等）を用いて独自に学習・推論パイプライン（または推論サーバー）を構築・運用する必要があります。
*   **推奨度**: **【非推奨】** - モデルのホスティングコストや保守コストが高く、本プロジェクトのフェーズに合わないため。

### (iii) LLMベースのアノテーション (GPT-4o / Claude 3.5 Sonnet)
*   **精度・特徴**: Few-shot Prompting（数例の具体例を提示）とChain-of-Thought（思考プロセスの記述）を組み合わせることで、LLMにDisfluencyの構造（Reparandum, Interregnum, Repair）を理解させ、抽出とクレンジングを同時に行わせる手法です。意味的な言い直しの判定においては現行最高峰の精度を誇ります。
*   **実装難度**: 低〜中。プロンプトエンジニアリングとJSON構造化出力APIの利用のみ。
*   **推奨度**: **【推奨】** - 高精度であり、インフラ保守不要。内容評価のプロセスと統合しやすいため。

## 4. アノテーション/中間表現の設計
【推測・提案】
解析・スコアリング（フィラー率算出）と、後段のLLM内容評価の双方に使いやすい「二層表現（Clean層 ＋ Disfluencyメタデータ層）」のJSONスキーマを提案します。

```json
{
  "original_text": "えーと、研究室では、えーと、…そこでは、えーと、シミュレテッドアニーリング、漁師、漁師、えーと、漁師なんとかシミュレーションを、えーと、専門で行っておりました。",
  "clean_text": "研究室では、そこでは、量子シミュレーションを専門で行っておりました。",
  "disfluencies": [
    {
      "type": "filler",
      "text": "えーと、",
      "count": 5
    },
    {
      "type": "repetition",
      "text": "漁師、漁師、",
      "count": 1
    },
    {
      "type": "repair",
      "reparandum": "シミュレテッドアニーリング、",
      "repair": "量子シミュレーション"
    }
  ],
  "metrics": {
    "total_chars": 76,
    "disfluency_chars": 39,
    "disfluency_rate": 0.51
  }
}
```
※文字単位の厳密なSpan（`"start": 0, "end": 4`等）はLLMのハルシネーション（数え間違い）が起きやすいため、実用上は「テキストの抽出」「カテゴリごとの出現回数/文字数」に留め、ロジック側で指標化するアプローチが安定します。

## 5. 機能語の曖昧性問題
【確認済み】
「その/あの/まあ/なんか/こう」といった語は、正規用法（指示詞や副詞としての機能）と、フィラー（思考の場つなぎ、意図的な曖昧性の付与）の双方の用法があり、語義曖昧性解消（WSD: Word Sense Disambiguation）の典型的な難所です。

*   **単純辞書マッチの限界**: UniDic等の形態素解析は主に「語の形」で品詞を引くため、「あの（連体詞）」として解析されたものが文脈上フィラーである場合、辞書ベースのみで誤検出・見逃しを防ぐのは困難です。
*   **先行研究の知見**: 文脈（前後関係、構文木の破綻）から判断する必要があります。近年はTransformerベースのモデル（BERTやLLM）が、Attention機構により前後の文脈情報を加味して「ここは言い淀みとしての『あの』である」と高精度に見分けることが実証されています。

## 6. 我々のユースケースへの推奨最小構成（推奨パイプライン）
面接評価ツール（フィラー率の指標化 ＋ 内容のLLM評価）における、精度・コスト・実装容易性のバランスを取った推奨パイプラインは以下の通りです。

**ステップ1: 形態素解析によるベースライン判定（高速・低コスト）**
*   **手法**: Whisper出力（verbatim）に対し、MeCab + UniDicを適用。
*   **役割**: 明らかな「感動詞-フィラー」（えー、あー、えーと 等）をカウントし、基礎的なフィラー率の暫定スコア（computed score）を算出します。

**ステップ2: LLMによる高度な非流暢性抽出とClean層の生成（高精度）**
*   **手法**: 内容評価を行うLLM（Claude/GPT）のプロンプト内に、Disfluencyの除去と抽出タスクを組み込みます。
*   **プロンプト方針**: 
    1. 【Few-shot】「繰り返し」「言い直し」「曖昧なフィラー（なんか/まあ）」の具体例を提示。
    2. 【出力形式】上記「4.」のようなJSON形式で、`disfluencies`のリストと、完全に読みやすく整えられた`clean_text`を出力させる。
*   **役割**: 形態素解析では取れない「繰り返し」「言い直し」「曖昧な機能語」を特定。得られた`clean_text`を、そのまま「構成力」「論理性」の内容評価タスクへとパイプラインで流します。

**決定論とLLMの役割分担まとめ**
*   **決定論（形態素）**: 高速な単純フィラーカウント（ベースライン指標）。
*   **LLM**: 複雑な非流暢性の検出、曖昧性解消、クリーンテキスト生成、および内容評価。この二段構えにより、インフラの複雑化（BERTの自己ホスト等）を避けつつ、最高クラスのアノテーションと評価パイプラインを実現できます。
