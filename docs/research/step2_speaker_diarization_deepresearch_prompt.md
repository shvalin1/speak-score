# ディープリサーチ用プロンプト：日本語1対1面接音声の話者分離（speaker diarization）

> 用途: Gemini / OpenAI / Perplexity 等の Deep Research に投入する調査依頼プロンプト。
> 目的: 「就活面接（および管理栄養士の面談）の音声を、応募者と面接官に分離し、応募者の発話だけを
> 評価対象にする」ための **CPU Cloud Run で出荷可能 or API 利用** の話者分離手法を、実在の OSS/モデル/API の
> 具体名・URL・ベンチマーク数値・料金・データ保持方針つきで確定する。

# 役割と目的

あなたは音声処理（speech processing）と話者ダイアライゼーション（speaker diarization）の実務・調査に
長けた専門リサーチャーです。**日本語の1対1の面接／面談音声（短尺・モノラル）を「誰がいつ話したか」で
分離し、特定の話者（＝評価対象＝応募者/ユーザー）の発話だけを取り出す手法**について、実際の事例や
実装可能な OSS / 事前学習済みモデル / クラウド API を、ソース（URL）・ベンチマーク・ライセンス・料金・
データ保持方針とともに調査・比較し、本ユースケースへの推奨構成を根拠付きで報告してください。

# 背景（このプロジェクトの文脈・前提知識として使ってよい）

- 動画面接スコアリング製品。動画→音声抽出→OpenAI Whisper で文字起こし→音声特徴量（librosa）と LLM で
  「話し方」と「内容/構成」を評価する。フィラー率・言い淀み・話速・抑揚などを定量化するのが差別化点。
- **いま話者分離を入れたい理由**: 実際の面接は「面接官の質問」＋「応募者の回答」が交互に来る。現状は全発話を
  一括で文字起こし・評価しているため、面接官の発話やフィラーまで応募者の評価に混入する。**応募者の発話だけ**を
  抽出して disfluency 検出・内容評価・話速/無音計測に渡したい。文字起こしスキーマには既に
  `speaker: str | None` フィールドがあり、各セグメントに話者ラベルを付ける想定。
- 同じ仕組みを、インターン先（食事指導アプリ）の「管理栄養士とユーザーの面談を文字起こし→評価・要約」にも
  転用予定。要件は話者分離・タイムスタンプ・（将来）感情解析。**汎用的な1対1日本語会話の話者分離**として
  使えると望ましい。

# 厳守すべき技術制約（重要・ここが結論を左右する）

1. **デプロイ環境＝CPU のみの Cloud Run**。GPU は無い。コンテナリソースは現状 **2 vCPU / 2 GiB RAM**、
   `containerConcurrency=1`、リクエスト timeout 1800 秒。→ **GPU 前提のモデルは不可**。
   （社内決定で pyannote の自前ホストは「GPU 前提・CPU Cloud Run で OOM/激遅」として GKE+GPU の後フェーズに
   後送り済み。本調査は「CPU で現実的に回る自前モデル」か「外部 API」のどちらかに収束させたい。）
2. **入力音声は WAV・モノラル・16 kHz**（Whisper 用に ffmpeg でダウンミックス済み）。
   → **マルチチャンネル/ステレオを使ったチャンネル分離は使えない**（音響的な話者分離が必須）。
3. **音声長は最大5分程度**（面接1問〜数問の短尺）。話者数は **多くは2人（面接官＋応募者）**だが、
   **自己録画で実質1人**のケースや、まれに**複数面接官**もありうる。話者数が未知でも頑健に動いてほしい。
4. **出荷単位の制約**: アプリ層エンジニアが触れるのは「backend コンテナイメージの再ビルド（Python 依存追加）」
   まで。新しい外部 API のキー管理（Secret Manager）やマシンリソース増強は別担当の作業になる。
   → 外部 API 案を出す場合は「新規キーが要る」ことを明示し、自前 OSS 案は「pip で入る Python 依存と
   モデルサイズ/RAM/CPU 実行時間」を必ず添える。
5. **プライバシー**: 面接/面談音声は PII。外部 API に送る場合は **学習オプトアウト・データ保持・ZDR（Zero
   Data Retention）の有無**を必ず調べる。
6. **既存の文字起こしは OpenAI Whisper**（whisper-1 の verbose_json でセグメント＋`no_speech_prob`、フィラー保持
   prompt＋`temperature=0`）。話者分離はこの Whisper 出力と**両立・整合**させたい（diarization を別に走らせて
   Whisper のセグメント/語に話者ラベルを貼る、もしくは文字起こしごと差し替える、の両案を比較）。

# すでに把握している事（重複説明は不要。これらを「裏取り・深掘り・反証」してほしい）

- 話者分離の一般パイプライン: VAD（音声区間検出）→ 話者埋め込み（speaker embedding）→ クラスタリング
  （AHC/スペクトラル）→ （任意で）再セグメンテーション／オーバーラップ処理。
- 代表的 OSS: pyannote.audio、NVIDIA NeMo（MSDD / Sortformer）、SpeechBrain、3D-Speaker、WeSpeaker、
  diart（オンライン）、silero-VAD / webrtcvad。WhisperX は pyannote を内部利用。
- 評価指標は DER（Diarization Error Rate、miss+false alarm+confusion）。

# 調査してほしい論点（各論点で参照できる実際の事例やソース情報・数値を付す）

1. **クラウド話者分離 API（新規キーありきの案）**
   - Google Cloud Speech-to-Text（v2 の speaker diarization）、AWS Transcribe、AssemblyAI、Deepgram、
     Speechmatics、Rev.ai、Azure Speech（conversation transcription）等について、
     **(a) 日本語対応の有無と品質、(b) 話者分離精度（公表 DER や事例）、(c) 文字起こしも兼ねるか（Whisper を
     置き換えられるか）、(d) 料金（分単価）、(e) データ保持/学習オプトアウト/ZDR、(f) 話者数指定の要否、
     (g) 最大音声長・レイテンシ** を一覧化。
   - 「diarization だけ」を提供する API があるか（文字起こしは Whisper のままにしたい場合）も調べる。

2. **CPU で現実的に動く自前（OSS）話者分離**
   - pyannote.audio を **CPU のみ**で動かした場合の実測（5分・16kHz mono を 2 vCPU/2 GiB で処理する所要時間と
     ピークメモリ）。OOM するのか、遅いだけで回るのか、量子化/軽量化で実用域に入るか。HuggingFace のモデル
     ゲーティング/ライセンス（pyannote/speaker-diarization-3.x の利用条件）も明記。
   - 軽量代替: silero-VAD/webrtcvad ＋ ECAPA-TDNN 等の埋め込み（SpeechBrain / WeSpeaker / 3D-Speaker）＋
     クラスタリングを **CPU で**組む構成。各モデルのサイズ・RAM・CPU 実行時間・日本語話者での精度。
   - NeMo（MSDD / Sortformer）や reverb-diarization 等の **CPU 実行可否**と精度/コスト。
   - 各手法に **推奨度（採用候補/参考/非推奨）** と根拠（精度・実装/保守コスト・CPU Cloud Run 適性・ライセンス）。

3. **「VAD 簡易分離」で足りるケースの見極め**
   - 自己録画で**実質1話者**の音声が多い場合、本格 diarization をやらず **VAD ＋「最も長く話している話者＝
     評価対象」前提**や**短い割り込み（相槌/質問）の除去**で十分な場面はどこか。簡易分離が破綻する条件
     （話者の声質が近い、オーバーラップ、面接官が長く話す等）を具体的に。

4. **評価対象話者（応募者）の自動特定**
   - 分離後の複数クラスタから「どれが応募者か」を決める実務手法: 累積発話時間が最長、質問の後に話す、
     音量/位置、あるいは**話者照合（speaker verification）**で既知サンプル（冒頭 N 秒や別録音）と
     マッチング。各手法の信頼性と失敗例。
   - エンロールメント（ユーザーに数秒の声サンプルを出させる）を前提にすると精度がどれだけ上がるか。

5. **Whisper 出力との統合（アライメント）**
   - diarization の話者区間と Whisper のセグメント/語タイムスタンプを突き合わせて
     `TranscriptSegment.speaker` を埋める実装パターン（WhisperX 方式・語単位割当・多数決等）と既知の落とし穴
     （境界ズレ、オーバーラップ、相槌、短発話の取りこぼし）。語単位タイムスタンプが要るか（whisper-1 は
     既定でセグメント単位）。

6. **日本語・短尺・少人数での精度の実態**
   - 日本語会話/面接系の話者分離ベンチマークやコーパス（CSJ の話者情報、CALLHOME Japanese、名大会話コーパス等）と
     公表 DER。2話者・5分・声質が近い・オーバーラップ少 という本条件での現実的な精度見込み。
   - 話者数が未知/1人のときの誤分割（1人を2人に割る）の頻度と対策。

7. **本ユースケースへの推奨アーキテクチャ（結論）**
   - 制約（CPU Cloud Run の Python 依存だけで出荷 ／ もしくは外部 API＋新規キー）を満たす中で、
     **ハッカソン向け最小構成**（短期・完走優先）と **発展版**（Phase3 GPU pyannote、Eatas 転用）を分けて提案。
   - 「diarization 専用 OSS/API ＋ 既存 Whisper」 vs 「文字起こし＋話者分離を一体で行うクラウド API（Whisper 置換）」の
     比較表と最終推奨。最小実装で踏むべき手順と、想定される落とし穴も。
